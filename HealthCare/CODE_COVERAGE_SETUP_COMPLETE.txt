## 📊 What The Report Shows

### Coverage Percentages
- **Line Coverage** - % of code lines executed by tests
- **Branch Coverage** - % of if/else branches tested  
- **Method Coverage** - % of methods called by tests

### Color Coding
🟢 Green (90-100%) = Excellent coverage
🟡 Yellow (60-89%) = Good coverage
🔴 Red (<60%) = Needs more tests

### File-By-File Breakdown
Shows coverage for each source file:
- Green sections = covered by tests ✅
- Red sections = not covered ❌
- Orange sections = partially covered ⚠️

## 📋 Common Commands

### Generate Coverage Only (No Report)
```bash
dotnet test /p:CollectCoverage=true
```

### Generate with File Output
```bash
dotnet test /p:CollectCoverage=true /p:CoverletOutputFormat=cobertura /p:CoverletOutput=mycoverage.xml
```

### View Specific Test Class Coverage
```bash
dotnet test --filter TestClass=DepartmentServiceTests /p:CollectCoverage=true
```

### Set Minimum Coverage Threshold
```bash
# Fail if coverage drops below 70%
dotnet test /p:CollectCoverage=true /p:Threshold=70
```

### Run Script (Easiest!)
```bash
.\run-code-coverage.ps1
```

## 💡 Tips for Better Coverage

### ✅ Do
- Focus on business logic
- Test error scenarios
- Test edge cases
- Write clear test names
- Use meaningful assertions

### ❌ Don't
- Obsess over 100% coverage
- Test trivial code (getters/setters)
- Write tests just for coverage
- Ignore test quality for numbers

### 🎯 Balance
- Aim for 80%+ overall
- 85%+ for critical code
- Quality over quantity
- Use coverage as a guide
